内卷化

1 什么是内卷化

概念：廉价劳动力过剩，从而无法带动技术革新

2 程序员“内卷化”

  1. 码农时代

  2. 开源时代

    • 强调整合，新技术学习

    • 工资倒挂：老员工维护老代码，新人掌握了新的技术框架很容易替换掉老员工

  3. 加班文化：

    • 开始大家加班了为了提高绩效，后来大家为了得到高绩效都采用加班来表达“态度”，慢慢加班成为了文化，程序员自己毁掉了自己的工作环境，由于长期加班，自己的思考变得迟钝，不再有更多的创新，不再想提供工作效率的方法，因为“加班”就可以了

如何不被”内卷化”

1) Stay Hungry, Stay Foolish
   * 保持好奇心，持续学习
   * 走出自己的舒适区
2) 工程能力：不能只会编码，要能部署，调试，打包等
   * 架构：学习业务知识，学习架构思维
   * 规范：流程规范，操作规范等
   * 管理：自己的日常管理，同事的协同管理
   * 排错：解决问题的嫩体力
3) 学会思考：
   * 高效工作，避免加班

编程范式

1 过程式编程

面向过程可以理解是做事情的动嘴进行分解多个步骤，所以有if/while 这类语法支撑，走不通的分支步骤

1 函数式编程

函数式编程是一种过程抽象的思维，就是对当前的动作去进行抽象，关注得是动作

函数式相比于过程式，更强调结果而非过程，所以函数式编程语言中不需要if/while这类语法，而是模式匹配，递归调用等。

2 面向对象编程

面向对象可以理解为对数据的抽象，比如把事物抽象成为一个对象，关注的是数据

面向对象

面向对象设计原则

  • solid

    单一职责:所有设计原则的基础和解决问题的思路

    开闭原则：设计的终极目标，应用开闭原则的关键则在于抽象

    里式替换原则：强调子类替换父类的正确性,它用来帮助实现开闭原则

    接口隔离原则：帮助实现里式替换原则

    依赖倒置：过程式设计与面向对象设计的分水岭，同时被用来指导接口隔离原则

         应用依赖倒置的关键在于“抽象” 和 “归纳”
     分析不变的部分(高层的，抽象的逻辑)，和容易变的部分(底层的，细节的逻辑)，把相对稳定的高层逻辑加以抽象，让它不再依赖于具体的底层逻辑

    面向过程与面向对象

    面向过程

    1.数据结构+算法
    2.数据本身无生命

    面向对象

    1.数据变得有生命，有血有肉
    2.

    形象比喻：

    • 面向过程：一堆东西扔到一个大柜子里
    • 面向对象：每样东西封装到一个小柜子里，最后放到大柜子里

windows权限

Windows访问控制概述:

Windows访问控制中最主要的部分：访问令牌（Access Token）和安全描述符（Security Descriptor），
window通过查看访问者AT与被访问对象SD中的内容来确定访问者是否能访问对象。

Windows访问控制概念介绍：

  • SID：Secure Identifier(安全标识符)，每个用户和账户组都有一个唯一的SID（通常情况下唯一）。它是标识用户、用户组和计算机账户唯一的号码，由计算机名、当前时间、当前用户态线程的CPU耗费时间三个参数来确定。
  • Access Token:与特定的windows账户关联，账户环境下启动的所有进程都会获得该令牌的副本，进程中的线程默认获得这个令牌。由关联账户的SID、当前登录账户所属组的SID列表、受限制的SID列表、当前登录账户与所属组的Privilege列表组成。
  • Security Descriptor:安全描述符，与被访问对象关联。由对象所有者的SID、属组SID、DACL、SACL组成。DACL（随机访问控制列表）是SD最重要的一部分，描述允许或拒绝特定用户或组的某些访问权限，它包含零个或多个访问控制实体（ACE，Access Control Entry）。
  • ACE：访问控制实体，用于指定特定用户/组的访问权限，由SID、SIZE、Type、Access Mask、Inheritance/Audit Flags组成。

Windows访问控制流程图：

当一个线程尝试去访问一个对象时，系统会检查线程持有的令牌以及被访问对象的安全描述符中的DACL。
如果安全描述符中不存在DACL，则系统会允许线程进行访问。
如果存在DACL，系统会顺序遍历DACL中的每个ACE，检查ACE中的SID在线程的令牌中是否存在。
以访问者中的User SID或Group SID作为关键字查询被访问对象中的DACL。
顺序：先查询类型为DENY的ACE，若命中且权限符合则访问拒绝；
未命中再在ALLOWED类型的ACE中查询，若命中且类型符合则可以访问；以上两步后还没命中那么访问拒绝。

如何沟通

###沟通是一种语言表达。表达背后反映的是一个人的思考方式

##丢掉自我为中心

  • 不要反问（强行拉别别人到你的价值观和判断体系，让人不舒服）

  • 换位思考
    错误：我都说的这么明白了，你怎么还听不清楚
    正确：我讲的足够清楚吗？ 潜台词如果不清楚是我的责任

##保持谦逊

  • 别人告知或询问
    回答：真棒，加油
    回到2：请稍等，我还需要几分钟

##黄金思维

  • why how what
    大众模式: what -> how -> why
    非凡模式：why -> how -> what

##金字塔原则 (帮助汇报困难户)

  • 基本结构：
    结论先行，以上统下，归类分组，逻辑递进
    先重要后次要，先总结后具体，先框架后细节，先结论后原因，先结果后过程，先论点后论据

    例：老板，我觉得这个方案最优，第一, xxx 第二 xxx 第三 xxx
    具体方案细则如下： 目标是xx, 计划xxx， 风险xxx ，规避风险方案xxx

##信息传递

  • 主动制定deadline
  • 主动汇报工作进展
  • 保持信息的沟通

npm使用

###简介

  • NPM是随同NodeJS一起安装的包管理工具
  • 从NPM服务器下载第三方包到本地使用
  • 从NPM服务器下载并安装别人编写的命令行程序到本地使用
  • 允许用户将自己编写的包或命令行程序上传到NPM服务器供别人使用
  • npm 由网站、注册表（registry）和命令行工具 (CLI)三部分组成：

网站 是开发者查找包（package）、设置参数以及管理 npm 使用体验的主要途径。
注册表 是一个巨大的数据库，保存了每个包（package）的信息。
CLI 通过命令行或终端运行。开发者通过 CLI 与 npm 打交道。

DDD

##软件核心复杂性应对之道

  1. 改变传统的数据库建模方法
  2. DDD先对业务领域进行建模，建立领域模型，根据领域模型驱动代码设计。合理运行面向对象高内聚低耦合设计要素，降低系统业务复杂度，系统具有更好的扩展性，更好的应对多变的业务需求

##领域 Domain

什么是领域？
答:一个领域就是一个问题域，统一个领域问题域相同

确定领域的作用是什么？
答: 驱动系统所属的领域，系统的核心业务即要解决问题及问题的边界基本就确定了。

例子：
陌生人设计领域，包含聊天，用户推荐，朋友圈等核心业务。主要是这个领域，一般都会相同的核心业务，他们解决问题的本质是一样的，就是交友

  • 社交领域：关键问题是用户推荐，聊天
  • 电商领域：关键问题是购物，订单，物流

##什么是子域？
答：复杂的领域拆分多个子域

子域的分类

  1. 核心域
  2. 支撑子域
  3. 通用子域

列子：社交领域的划分

  • 核心域：用户推荐 聊天
  • 支撑子域：客服 反垃圾
  • 通用子域：消息推送

##驱动设计

  1. 以领域边界，分析核心问题
  2. 设计对应的领域模型
  3. 通过领域模型驱动代码设计的实现
  4. 设计合理的分层和解耦

##概念总结

  • 领域就是问题域
  • 模型驱动的思想: 通过建立领域模型来解决核心问题
  • 领域模型的目标: 针对领域的核心问题，不是所有问题
  • 领域模型设计: 考虑抽象性 通用 复用
